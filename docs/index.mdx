Yieldo is a cross-chain deposit aggregator that lets users deposit into yield vaults from any supported chain and token. The API handles routing, bridging, and vault deposits with no fees to the user. An on-chain attribution system tracks deposits for revenue share with vault partners.

## Base URL

```
https://api.yieldo.xyz
```

## How It Works

1. **Browse vaults** - Query available vaults across chains and protocols
2. **Get a quote** - Submit your source chain, token, and amount to get a deposit quote with route options
3. **Build the transaction** - Submit the quote data to get a ready-to-send transaction
4. **Send & track** - Send the transaction and monitor the deposit progress

## Key Concepts

### Attribution-Only Router (V3.2.0)

Yieldo's deposit router is non-custodial — it pulls tokens from the caller, deposits them into the vault, and emits a `Routed` event for on-chain attribution. No fees are deducted. 100% of the user's deposit goes into the vault.

V3.2.0 (live on every chain — Ethereum, Base, Arbitrum, Optimism, HyperEVM, Katana) adds **`depositForAvailable`**, which pulls `min(allowance, balance)` from `msg.sender` instead of an explicit amount. This is what cross-chain composer routes call: the bridge receiver (Across, Stargate, etc.) approves the router for the post-bridge amount and the router sweeps whatever actually arrived. Eliminates the "calldata says X, bridge delivered X-1, deposit reverts" failure mode entirely.

V3.2.0 also handles ERC-4626 vaults whose `deposit()` returns void (some older OZ forks) via a `balanceOf` delta fallback.

V3.1.1 carried over: depositFor entry point is **open to any caller** — any address can call with `user=<anyone>`, so the composer flow works across every LiFi bridge-receiver variant (ReceiverAcrossV3/V4, ReceiverStargateV2, ReceiverChainflip, per-chain Executors) without needing a router upgrade whenever LiFi adds a new bridge. Funds are always pulled from `msg.sender`, so the caller can only ever spend their own balance.

V3.1.0 shipped an 8-arg `depositFor` overload with `minSharesOut` for on-chain slippage protection (7-arg form kept as compat shim), and added `shares` to the `Routed` event.

Each deposit carries a `partnerId` and `partnerType` for attribution — identifying which wallet, creator, or integration referred the user. This is used for revenue share tracking with vault partners.

See [Deposit Router Contracts](/guides/deposit-router-contracts) for the full interface and per-chain addresses.

### Cross-Chain Routing

For cross-chain deposits, Yieldo uses [LiFi](https://li.fi/) to bridge and swap tokens. Users can choose from multiple bridge routes (like Jumper.exchange), with the best route auto-selected.

Two flows are supported:
- **Single-step (LiFi Composer)** — used for Morpho / ERC-4626 vaults. LiFi bridges and calls `router.depositForAvailable(...)` on the destination in one tx. Any bridge receiver can make the call (V3.1.1 — no whitelist), and the router sweeps the actual delivered amount via `min(allowance, balance)` (V3.2.0 — bridge underdelivery can't revert the deposit).
- **Two-step** — used for vault types that aren't composable with the bridge (Midas, Veda, Custom, IPOR, Lido). Step 1 bridges tokens to the user's wallet on the destination chain; step 2 is a same-chain `depositFor` call the user signs themselves.

### Vault Adapter Pattern

The router supports any vault type through a pluggable adapter system. Standard ERC-4626 vaults work out of the box. Custom vaults (Midas, Veda, Lido, etc.) use adapters that can be deployed without upgrading the router contract.

### Supported Vault Types

Each vault has a `type` field that determines how the router dispatches deposits. The router handles routing internally — integrators only pass `isERC4626` via `partner_type`/`partner_id`.

| Type            | Description                                     | Examples                                       |
| --------------- | ----------------------------------------------- | ---------------------------------------------- |
| `morpho`        | Standard ERC-4626 (`deposit(amount, to)`)       | Steakhouse USDC, Gauntlet WETH Prime, Moonwell |
| `veda`          | Veda BoringVault via Teller contract            | Veda Liquid ETH/BTC/USD, PlasmaUSD, Lombard    |
| `midas`         | Midas Issuance Vault via `depositInstant`        | mTBILL, mAPOLLO, mHYPER, Midas RE7             |
| `custom`        | `syncDeposit(amount, to, address(0))`           | 9Summits Flagship ETH, Hyperbeat vaults        |
| `lido`          | Lido Earn SyncDepositQueue                       | Lido Earn USDC/ETH                             |
| `ipor`          | IPOR-specific ERC-4626 with quirks               | IPOR vaults                                    |
| `accountable`   | Accountable protocol ERC-4626                    | Hyperithm Delta Neutral, Yuzu Money            |
| `unsupported`   | Router doesn't support — blocked in quote API    | Paused Midas vaults, unverifiable Mellow proxies |

### Revenue Model

Yieldo charges **no fees to users**. Revenue comes from revenue share agreements with vault curators and protocols. Every deposit routed through Yieldo is attributed on-chain via the `Routed` event, providing provable attribution for invoicing.

## Quick Example

```bash
# 1. List vaults on Base
curl https://api.yieldo.xyz/v1/vaults?chain_id=8453

# 2. Get a quote to deposit 1000 USDC from Arbitrum into a Base vault
curl -X POST https://api.yieldo.xyz/v1/quote \
  -H "Content-Type: application/json" \
  -d '{
    "from_chain_id": 42161,
    "from_token": "0xaf88d065e77c8cC2239327C5EDb3A432268e5831",
    "from_amount": "1000000000",
    "vault_id": "8453:0xbeefe94c8ad530842bfe7d8b397938ffc1cb83b2",
    "user_address": "0xYourAddress"
  }'

# 3. Build the transaction
curl -X POST https://api.yieldo.xyz/v1/quote/build \
  -H "Content-Type: application/json" \
  -d '{
    "from_chain_id": 42161,
    "from_token": "0xaf88d065e77c8cC2239327C5EDb3A432268e5831",
    "from_amount": "1000000000",
    "vault_id": "8453:0xbeefe94c8ad530842bfe7d8b397938ffc1cb83b2",
    "user_address": "0xYourAddress"
  }'
```
